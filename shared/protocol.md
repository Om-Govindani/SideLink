# SideLink Communication Protocol Specification (V1)

This document defines the network ports, data framing, security handshake, and packet structures used for communication between the macOS Host and the Android Client.

---

## Network Architecture & Port Allocation

All communication takes place over a local network (Wi-Fi or USB tethering). The macOS host binds to two TCP sockets:

| Port | Protocol | Purpose | Description |
|---|---|---|---|
| **5230** | TCP | **Control Channel** | Exposes pairing handshake, configuration settings, and client input events (touch, mouse). |
| **5231** | TCP | **Video Stream Channel** | Delivers framed hardware-encoded H.264 NAL units from macOS to Android. |

*Both TCP sockets are configured with `TCP_NODELAY = 1` to disable Nagle's algorithm and ensure immediate packet delivery.*

---

## QR Code Connection URI Format

The macOS app displays a QR code encoding a connection string with the following schema:

```
sidelink://connect?ip=<IP_ADDRESS>&port=<PORT>&secret=<PAIRING_SECRET>
```

* **`ip`**: The local IPv4 address of the macOS host (e.g., `192.168.1.15`).
* **`port`**: The control port (normally `5230`).
* **`secret`**: A cryptographically random, hex-encoded 32-character string generated by the macOS Host upon startup. It is single-use and secures the pairing.

---

## Connection & Handshake Flow

1. **Host Setup**:
   - The Mac app starts, creates a new random `pairingSecret`, and listens on ports `5230` and `5231`.
   - The Mac app displays the QR code.
2. **Client Scan**:
   - Android scans the QR code and parses the host details.
   - Android opens a TCP connection to the Mac's control port (`5230`).
3. **Handshake**:
   - The Client sends a `Handshake Request` packet containing the `pairingSecret`.
   - The Host verifies the secret.
     - **If valid**: Host responds with `Handshake Response (Success)`. Host then accepts a TCP connection on the stream port (`5231`), starts capturing, and initiates the H.264 encoding stream.
     - **If invalid**: Host responds with `Handshake Response (Failure)` and closes the connection.

---

## Control Packet Structure

All control packets sent over port `5230` utilize a unified binary structure:

```
+---------------------------+------------------------+-------------------------------+
|  Packet Length (4 Bytes)  | Packet Type (1 Byte)   |       Payload (Variable)       |
|  UInt32 (Big Endian)      | UInt8                  |                               |
+---------------------------+------------------------+-------------------------------+
```

### Packet Types

#### `0x01`: Handshake Request (Client -> Host)
* **Payload**: UTF-8 encoded pairing secret string.
* *Example structure*: `[Length: 33] [Type: 0x01] [Payload: 32 bytes of ASCII secret]`

#### `0x02`: Handshake Response (Host -> Client)
* **Payload**: `0x00` (Success) or `0x01` (Invalid Secret).
* *Example structure*: `[Length: 2] [Type: 0x02] [Payload: 0x00]`

#### `0x03`: Configure Display (Client -> Host)
Sent by the client immediately after successful handshake to request display parameters.
* **Payload**:
  - Width: 4 Bytes (UInt32) - Preferred display width (e.g., `1920`)
  - Height: 4 Bytes (UInt32) - Preferred display height (e.g., `1080`)
  - FPS: 1 Byte (UInt8) - Preferred frame rate (e.g., `60`)

#### `0x04`: Input Event (Client -> Host)
Transmits client input actions.
* **Payload**:
  - Touch Type: 1 Byte (UInt8)
    - `0x00`: Pointer Down
    - `0x01`: Pointer Move (Drag)
    - `0x02`: Pointer Up
    - `0x03`: Secondary Click (Right-click equivalent / Two-finger tap)
  - X Coordinate: 4 Bytes (Float32, Big Endian) - Normalized coordinate `[0.0 - 1.0]`
  - Y Coordinate: 4 Bytes (Float32, Big Endian) - Normalized coordinate `[0.0 - 1.0]`

---

## Video Stream Framing

Instead of raw H.264 byte stream scanning (which demands high CPU overhead to find start codes), the stream connection over port `5231` uses length-prefixed NAL units:

```
+---------------------------+--------------------------------------------------------+
|   Frame Length (4 Bytes)  |                   H.264 Payload                        |
|   UInt32 (Big Endian)      |                   (SPS / PPS / I-Frame / P-Frame)      |
+---------------------------+--------------------------------------------------------+
```

1. **Host Encoder**:
   - VideoToolbox outputs encoded H.264 NALUs.
   - The host writes the size of the NALU as a 4-byte big-endian integer, followed immediately by the H.264 raw bytes (excluding Annex B start codes, or with start codes intact, but length-prefix is checked).
2. **Client Decoder**:
   - The client reads 4 bytes to find the packet size.
   - It reads the specified number of bytes, packs them, and queues the array directly into `MediaCodec.queueInputBuffer()`.
